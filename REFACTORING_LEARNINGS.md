# Learnings: When "Pure Refactorings" Aren't

## Incident Summary (June 6, 2026)

A seemingly safe refactoring broke Sonatype Maven Central publishing, causing release 3.1.1 to fail. The refactoring extracted inline POM configuration logic into top-level helper functions - a change that appeared purely structural with no behavioral differences.

## What Happened

### The Refactoring (Commit d149003a)

**Before:**
```kotlin
afterEvaluate {
    publishing.publications.withType<MavenPublication>().forEach {
        with(it) {
            pom.name.set(project.name)
            pom.licenses { /* ... */ }
            // All config inline
        }
    }
}
```

**After:**
```kotlin
afterEvaluate {
    publishing.publications.withType<MavenPublication>().forEach { configurePom(it) }
}

fun configurePom(publication: MavenPublication) {
    publication.pom.apply {
        name.set(project.name)  // ⚠️ Breaks Sonatype
    }
}
```

### The Failure

- `initializeSonatypeStagingRepository` failed with HTTP 403
- Sonatype returned HTML error page (not JSON)
- Same credentials worked in other repositories (testmints, jsmints)
- Generated POM files were byte-for-byte identical
- Reverting the refactoring fixed the issue immediately

## Key Lessons

### 1. Plugin DSL Context Matters

In Gradle precompiled script plugins (`.gradle.kts` in `src/main/kotlin`):
- Code within `with()` blocks has explicit receiver context
- Top-level functions may capture `project` references differently
- This affects how other plugins (like nexus-publish-plugin) interact with your configuration

**Implication**: Refactorings that only move code location can change semantics in DSL-heavy systems.

### 2. Generated Artifacts ≠ Runtime Behavior

The POM files generated by both versions were identical, yet one version failed to publish. The issue occurred during Sonatype API initialization, before any POMs were even generated.

**Implication**: Testing output artifacts alone is insufficient - you must test the full integration flow.

### 3. Integration Failures Can Be Silent

The refactoring:
- Passed all unit tests
- Generated correct POMs locally
- Only failed during actual Sonatype API interaction in CI

**Implication**: Some bugs only manifest in production-like environments with real external services.

### 4. HTTP Error Codes Can Mislead

The 403 error suggested an authentication/authorization problem, but:
- Credentials were valid (proven by other repos)
- The issue was in how the HTTP request was formed
- Root cause was in client code, not server permissions

**Implication**: Don't assume error codes always point to the obvious cause.

## Detection Strategies

### What Didn't Work
- ❌ Unit tests of POM generation
- ❌ Local build verification
- ❌ Linting/static analysis

### What Did Work
- ✅ Full release workflow in CI with real Sonatype credentials
- ✅ Comparing successful vs failed runs
- ✅ Git bisect to find the breaking commit
- ✅ Reverting and re-testing

### Future Prevention

1. **CI-based integration tests** - Release workflow caught this
2. **Careful code review** - Flag any changes to plugin DSL code
3. **Documentation** - Added warnings in code and docs
4. **Git history** - Preserve evidence for future debugging

## Broader Implications

### When is a Refactoring "Safe"?

Traditional definition: Code changes that preserve observable behavior.

But in plugin/DSL systems:
- Evaluation order matters
- Context capture matters
- External tool interaction matters
- "Observable behavior" includes HOW the system is configured, not just WHAT it produces

### Trust, But Verify

Even when:
- Code looks equivalent
- Tests pass
- Output is identical

...the integration might still break. Always:
- Test in production-like environments
- Monitor actual external interactions
- Keep rollback plans ready
- Document incidents for future learning
